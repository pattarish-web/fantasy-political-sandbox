# Fantasy World and Continuity Design

## Goal

ทำให้ระบบสร้างนิยายรองรับเผ่าพันธุ์ เวทมนตร์ ธาตุ และเผ่าใหม่ที่เปิดเผยตามเนื้อเรื่อง โดยรักษาความต่อเนื่องของไทม์ไลน์ ตัวละคร ความสัมพันธ์ และข้อมูลภาษาไทย

## Scope

- World Bible กลางสำหรับเผ่า ธาตุ สำนักเวท จุดอ่อน และราคาที่ต้องจ่าย
- race registry ที่แยกเผ่าซึ่งค้นพบแล้วออกจากเผ่าที่ซ่อนอยู่
- ข้อมูลตัวละครใหม่ใน `meta_data` เดิม เพื่อลดความเสี่ยง migration
- validation และ fallback สำหรับข้อมูล AI ที่ไม่ครบหรือขัดแย้ง
- timeline, character knowledge, relationship state และ continuity gate ก่อนบันทึกตอน
- prompt ของการจำลอง/เขียนตอนให้ใช้บริบทโลกเดียวกัน
- หน้าเว็บแสดงข้อมูลแฟนตาซีใหม่เป็นภาษาไทย
- regression tests สำหรับข้อมูลไม่ครบ เผ่าโผล่ผิดเวลา และความขัดแย้งของข้อมูล

## Invariants

1. ชื่อตัวละครและข้อมูลที่ผู้ใช้อ่านต้องเป็นภาษาไทยหรือคำอ่านไทยเท่านั้น
2. ตัวละครหนึ่งคนมีเวทหลักและธาตุเด่นได้อย่างละหนึ่งรายการ เว้นแต่ World Bible ระบุข้อยกเว้น
3. เวททุกชนิดต้องมีข้อจำกัดและต้นทุน
4. เผ่าใหม่ต้องมีเบาะแสหรือเหตุการณ์เปิดตัวก่อนสถานะ `discovered`
5. ตัวละครรับรู้เฉพาะข้อมูลที่อยู่ใน knowledge state ของตน
6. เหตุการณ์สำคัญต้องบันทึกผลกระทบถาวรและลำดับเวลา
7. ข้อมูลที่ AI ส่งไม่ครบต้องใช้ค่าเริ่มต้นที่ปลอดภัย ไม่ทำให้ workflow ล้มเหลวโดยไม่จำเป็น

## Data Flow

World Bible และ registry ถูกโหลดเข้าสู่ prompt ของการจำลองและ historian ทุกครั้ง ผลลัพธ์ AI ผ่าน schema normalization จากนั้นผ่าน timeline/knowledge/continuity validation ก่อนบันทึกลงฐานข้อมูลและ export ไปยังหน้าเว็บ หาก validation ไม่ผ่าน ระบบจะ retry แบบจำกัดหรือเติม fallback พร้อมบันทึก warning

## Acceptance Criteria

- สร้างตัวละครใหม่ได้แม้ AI ไม่ส่งฟิลด์แฟนตาซีครบ โดยมีค่าเริ่มต้นชัดเจน
- ไม่สามารถเปิดตัวเผ่าใหม่โดยไม่มีเบาะแสหรือเหตุการณ์รองรับ
- ตอนใหม่ไม่สามารถอ้างข้อมูลที่เกิดในอนาคตหรือข้อมูลที่ตัวละครยังไม่รู้
- การจำลองและการเขียนตอนใช้กฎเวท/เผ่าชุดเดียวกัน
- หน้าโปรไฟล์แสดงเผ่า ธาตุ เวท จุดอ่อน และต้นทุนเป็นภาษาไทย
- ชุดทดสอบเดิมทั้งหมดผ่าน และมี regression tests สำหรับ invariants ข้างต้น
